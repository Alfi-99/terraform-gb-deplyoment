# Soal: Desain dan Implementasi Arsitektur Cloud AWS dengan Blue/Green Deployment

## Latar Belakang

Sebuah perusahaan teknologi sedang dalam proses modernisasi infrastruktur digitalnya. Mereka ingin membangun sebuah sistem aplikasi web yang scalable, resilient, dan dapat di-deploy dengan aman tanpa downtime. Tim DevOps perusahaan tersebut diminta untuk merancang dan mengimplementasikan sebuah arsitektur cloud di atas AWS yang mengikuti praktik terbaik industry, menggunakan pendekatan Infrastructure as Code (IaC) dengan Terraform, serta memanfaatkan layanan-layanan AWS yang tersedia dalam paket Free Tier semaksimal mungkin.

Sistem yang akan dibangun harus mampu menangani proses deployment tanpa gangguan layanan kepada pengguna akhir. Untuk mencapai hal tersebut, tim memilih strategi **Blue/Green Deployment** — sebuah teknik di mana dua lingkungan identik dijalankan secara bersamaan, dan perpindahan traffic antar versi dilakukan secara instan melalui konfigurasi load balancer, sehingga rollback dapat dilakukan dalam hitungan detik apabila terjadi masalah pada versi baru.

Selain itu, seluruh aktivitas sistem — mulai dari setiap request API yang masuk, proses deployment yang dilakukan, hingga error yang terjadi di dalam Lambda function — harus dicatat secara otomatis ke dalam database log terpusat menggunakan **Amazon DynamoDB**. Pencatatan log ini bertujuan untuk memudahkan audit, debugging, dan pemantauan kesehatan sistem secara real-time.

## Topologi Arsitektur

Arsitektur yang dibangun mengikuti gambar topologi berikut: sebuah **GitHub Repository** menjadi sumber kebenaran (single source of truth) dari seluruh kode aplikasi. Setiap perubahan kode yang di-push ke repository akan secara otomatis memicu **GitHub Actions** untuk menjalankan proses CI/CD — mulai dari testing, building, hingga packaging aplikasi menjadi sebuah file ZIP yang kemudian diunggah ke **S3 Bucket** khusus deployment artifacts.

Di sisi frontend, **AWS Amplify** terhubung langsung ke GitHub Repository dan secara otomatis melakukan build dan deploy setiap kali ada perubahan di branch yang dikonfigurasi. Amplify kemudian menampilkan antarmuka pengguna yang berkomunikasi dengan backend melalui URL **Application Load Balancer (ALB)** dan **API Gateway**.

Di sisi backend, **Application Load Balancer** berdiri di Public Subnet dan bertugas mendistribusikan traffic masuk dari internet. ALB ini memiliki dua Target Group: satu untuk environment **BLUE** dan satu untuk environment **GREEN**. Dengan mengubah bobot (weight) di listener ALB, tim dapat memindahkan seluruh traffic dari satu environment ke environment lainnya tanpa downtime. Kedua environment Elastic Beanstalk ini berjalan di **Private Subnet** yang terisolasi dari akses internet langsung.

**Elastic Beanstalk** menjadi platform utama yang menghosting aplikasi backend. Ia menyediakan kemudahan manajemen infrastruktur — dari provisioning EC2 instance, auto scaling, health monitoring, hingga deployment rolling — sehingga developer dapat fokus pada kode aplikasi. Untuk kebutuhan penyimpanan file yang perlu diakses bersama oleh semua instance, digunakan **EFS (Elastic File System)**, yang dapat di-mount ke semua EC2 instance layaknya sebuah shared hard disk jaringan. Selain itu, **S3 Bucket** kedua berfungsi sebagai object storage untuk menyimpan file-file yang diupload oleh pengguna aplikasi.

Ketika request API masuk melalui **API Gateway**, request tersebut diteruskan ke salah satu dari dua **Lambda Function** — satu untuk menangani operasi tulis (POST, PUT, DELETE) dan satu untuk menangani operasi baca (GET). Kedua Lambda function ini berjalan di dalam VPC agar dapat mengakses **RDS MySQL** yang berada di Private Subnet. Setiap request yang masuk dan keluar dari Lambda, beserta seluruh log aktivitas sistem, dicatat secara real-time ke tabel-tabel **DynamoDB**.

## Deskripsi Layanan yang Digunakan

Berikut adalah penjelasan lengkap untuk setiap layanan AWS yang digunakan dalam arsitektur ini beserta konfigurasi yang diperlukan:

### 1. VPC (Virtual Private Cloud)

VPC adalah fondasi jaringan dari seluruh arsitektur ini. Bayangkan VPC sebagai sebuah gedung kantor privat milik perusahaan yang terletak di dalam kawasan cloud AWS — di dalamnya, semua komponen infrastruktur bisa berkomunikasi satu sama lain secara aman dan terisolasi dari lingkungan luar. VPC yang dibangun menggunakan CIDR block `10.0.0.0/16`, yang memberikan ruang alamat IP yang cukup untuk ratusan resource. Di dalam VPC ini terdapat empat subnet yang dibagi ke dalam dua availability zone untuk ketersediaan tinggi: dua **Public Subnet** (masing-masing di AZ-a dan AZ-b) yang digunakan oleh Load Balancer dan NAT Gateway, serta dua **Private Subnet** (masing-masing di AZ-a dan AZ-b) yang digunakan oleh Elastic Beanstalk, RDS, Lambda, dan EFS. Agar Public Subnet dapat berkomunikasi dengan internet, dipasang sebuah **Internet Gateway**. Sementara itu, agar resource di Private Subnet dapat mengakses internet secara satu arah (misalnya untuk download package atau update), dipasang sebuah **NAT Gateway** di Public Subnet. Dua **Route Table** dikonfigurasi: satu untuk Public Subnet yang mengarahkan traffic ke Internet Gateway, dan satu untuk Private Subnet yang mengarahkan traffic ke NAT Gateway. Terakhir, dua **VPC Endpoint** bertipe Gateway ditambahkan untuk S3 dan DynamoDB agar traffic ke kedua layanan tersebut tidak perlu melewati internet, menghemat biaya dan meningkatkan keamanan.

### 2. Security Groups

Security Group adalah firewall virtual yang mengontrol traffic masuk dan keluar dari setiap resource. Dalam arsitektur ini, terdapat lima Security Group yang masing-masing memiliki aturan yang ketat sesuai fungsinya. **ALB Security Group** mengizinkan traffic HTTP (port 80) dan HTTPS (port 443) dari seluruh internet, karena ALB memang dimaksudkan sebagai titik masuk publik. **Elastic Beanstalk Security Group** hanya mengizinkan traffic masuk dari ALB Security Group pada port 80 dan 8080, memastikan tidak ada akses langsung ke EC2 instance dari luar. **RDS Security Group** hanya menerima koneksi MySQL (port 3306) dari EB Security Group dan Lambda Security Group, menjaga agar database tidak dapat diakses dari resource lain yang tidak berwenang. **Lambda Security Group** dikonfigurasi hanya dengan aturan outbound penuh agar Lambda dapat mengakses RDS, DynamoDB, dan service lainnya. **EFS Security Group** hanya mengizinkan koneksi NFS (port 2049) dari EB Security Group agar shared storage hanya dapat di-mount oleh EC2 instance Elastic Beanstalk.

### 3. IAM (Identity and Access Management)

IAM mengatur siapa (identitas) yang boleh melakukan apa (aksi) terhadap resource mana di dalam akun AWS. Prinsip yang diterapkan adalah **least privilege** — setiap identitas hanya diberikan izin minimum yang dibutuhkan. **IAM Role untuk Elastic Beanstalk Instance** memiliki izin untuk membaca dan menulis ke S3 bucket, menulis log ke DynamoDB, dan mengirim metrik ke CloudWatch. **IAM Role untuk Elastic Beanstalk Service** menggunakan managed policy AWS untuk enhanced health monitoring dan managed updates. **IAM Role untuk Lambda** memiliki izin yang lebih luas: akses ke DynamoDB untuk logging, akses S3 untuk file storage, izin untuk membuat dan menghapus network interface di VPC, akses ke Secrets Manager untuk mengambil kredensial database, serta izin menulis log ke CloudWatch. **IAM Role untuk GitHub Actions** menggunakan mekanisme OIDC Federation sehingga GitHub Actions dapat mengakses AWS tanpa perlu menyimpan long-lived access key, meningkatkan keamanan secara signifikan. Selain IAM Role, sebuah **AWS Secrets Manager Secret** dibuat untuk menyimpan kredensial database (username, password, host, port, nama database) secara terenkripsi — Lambda function mengambil kredensial ini setiap kali diinisialisasi, bukan dari environment variable yang bisa bocor di log.

### 4. S3 (Simple Storage Service)

Dua buah S3 bucket digunakan dalam arsitektur ini dengan tujuan yang berbeda. **Deployment Bucket** adalah gudang penyimpanan artifact deployment yang dihasilkan oleh GitHub Actions. Setiap kali ada push ke repository, GitHub Actions akan mem-build aplikasi, mem-package menjadi file ZIP, lalu mengunggahnya ke bucket ini dengan path yang mencerminkan versi dan warna environment (misalnya `deployments/v1.0.0-blue/app.zip`). Bucket ini dikonfigurasi dengan versioning enabled agar setiap versi artifact tersimpan dengan aman, enkripsi AES256 server-side, block public access sepenuhnya, dan lifecycle policy yang menghapus versi lama setelah 30 hari untuk menghemat biaya. **Application Storage Bucket** adalah tempat penyimpanan file yang diupload oleh pengguna aplikasi seperti gambar, dokumen, atau file lainnya. Bucket ini dikonfigurasi dengan CORS policy yang memungkinkan frontend mengupload langsung dari browser, enkripsi server-side, block public access, dan lifecycle policy yang memindahkan objek ke storage class Infrequent Access setelah 30 hari (lebih murah) dan menghapusnya setelah 365 hari.

### 5. EFS (Elastic File System)

EFS adalah layanan penyimpanan file terkelola yang dapat di-mount secara bersamaan ke banyak EC2 instance, layaknya sebuah shared network drive. Ini menjadi sangat penting dalam konteks Elastic Beanstalk yang bisa memiliki lebih dari satu instance berjalan bersamaan — tanpa EFS, file yang diupload user ke instance A tidak akan terlihat oleh instance B. EFS dikonfigurasi dengan performance mode `generalPurpose` yang cocok untuk beban kerja yang beragam, throughput mode `bursting` yang mengikuti pola penggunaan aktual dan lebih hemat biaya dibandingkan `provisioned`. Enkripsi at-rest diaktifkan untuk keamanan data. Lifecycle policy dikonfigurasi untuk memindahkan file yang tidak diakses selama 30 hari ke storage class Infrequent Access. **Mount Target** dibuat di setiap Private Subnet agar instance di kedua availability zone dapat mengakses EFS. Dua **Access Point** dibuat dengan kontrol akses berbeda: satu untuk direktori `/app-data` dan satu untuk direktori `/uploads`, dengan izin POSIX yang sesuai. **EFS Backup Policy** diaktifkan untuk perlindungan data otomatis.

### 6. Application Load Balancer (ALB)

ALB adalah komponen kunci yang membuat Blue/Green deployment bisa berjalan mulus. Ia berdiri di Public Subnet, bisa diakses dari internet, dan memiliki dua **Target Group** — satu bernama `blue-tg` yang berisi instance EC2 dari environment BLUE, dan satu bernama `green-tg` yang berisi instance dari environment GREEN. Setiap Target Group dikonfigurasi dengan health check ke endpoint `/health` setiap 30 detik — jika suatu instance gagal health check tiga kali berturut-turut, ia dikeluarkan dari Target Group secara otomatis. **Listener HTTP** pada port 80 dikonfigurasi dengan aturan **weighted forwarding**: ketika `active_color = "blue"`, weight untuk blue-tg adalah 100 dan green-tg adalah 0; ketika di-switch ke green, nilai tersebut dibalik. Perpindahan weight ini yang menjadi mekanisme Blue/Green switch — dapat dilakukan hanya dengan satu perintah `terraform apply -var="active_color=green"`. Sebuah Listener Rule tambahan dikonfigurasi untuk meredirect semua path `/api/*` ke URL API Gateway, sehingga frontend cukup berkomunikasi dengan satu base URL (ALB) tanpa perlu mengetahui URL API Gateway secara langsung.

### 7. Elastic Beanstalk (Blue/Green Environments)

Elastic Beanstalk adalah Platform-as-a-Service yang menyederhanakan deployment dan pengelolaan aplikasi di EC2. Dalam arsitektur ini, satu **EB Application** bernama `gb-app-prod-app` memiliki dua **Environment**: `gb-app-prod-blue` dan `gb-app-prod-green`. Kedua environment dikonfigurasi dengan pengaturan yang identik — platform Node.js 20 di Amazon Linux 2023, instance type `t2.micro` (free tier eligible), berjalan di Private Subnet, menggunakan Shared Load Balancer (ALB yang sudah dibuat sebelumnya), CloudWatch Logs streaming dengan retensi 7 hari, health reporting enhanced, dan managed updates minor setiap hari Minggu pukul 02.00 WIB. Perbedaan di antara keduanya hanya pada environment variable `DEPLOYMENT_COLOR` (bernilai `"blue"` atau `"green"`) dan `version_label` yang menunjuk ke versi aplikasi yang berbeda. Setiap environment mendapatkan environment variable penting seperti `DB_SECRET_ARN` untuk mengambil kredensial database, `S3_BUCKET` untuk storage, `DYNAMODB_LOG_TABLE` untuk logging, dan `API_GATEWAY_URL`. Deployment policy yang digunakan adalah **Rolling** dengan batch size 50% untuk meminimalkan downtime bahkan di dalam satu environment.

### 8. RDS MySQL

RDS adalah layanan database relasional terkelola yang menghilangkan kebutuhan administrasi database manual seperti patching, backup, dan failover. Database MySQL 8.0 di-deploy menggunakan instance class `db.t3.micro` yang masuk dalam Free Tier AWS (750 jam per bulan). Storage dikonfigurasi sebesar 20GB dengan tipe `gp2` — ini adalah batas maksimal storage gratis di Free Tier, sehingga autoscaling storage dinonaktifkan untuk menghindari biaya tak terduga. Multi-AZ dinonaktifkan karena akan menggandakan biaya. Database berada di Private Subnet yang sama dengan Elastic Beanstalk dan hanya dapat diakses melalui security group yang telah dikonfigurasi. Kredensial database tidak diinject langsung sebagai environment variable, melainkan disimpan di Secrets Manager dan diambil secara programatik oleh aplikasi saat pertama kali dijalankan. Parameter Group dikonfigurasi untuk mendukung karakter UTF-8 penuh (`utf8mb4`) agar mendukung karakter multibahasa termasuk emoji, serta batas koneksi yang sesuai kapasitas instance t3.micro. Backup otomatis dikonfigurasi dengan retensi 1 hari, dan Performance Insights diaktifkan dengan retensi 7 hari (gratis) untuk memudahkan analisis performa query.

### 9. DynamoDB (Log Storage)

DynamoDB dipilih sebagai penyimpanan log karena karakteristiknya yang sangat cocok untuk workload logging: write throughput tinggi, latensi sangat rendah, tanpa schema yang kaku (schemaless), dan Free Tier yang sangat dermawan (25GB storage, 25 WCU, 25 RCU). Tiga tabel DynamoDB dibuat dalam arsitektur ini. **Tabel `gb-app-prod-app-logs`** adalah buku besar utama yang mencatat semua aktivitas dari semua service — setiap log entry memiliki partition key `service_id` (nama service seperti `lambda-post`, `lambda-get`, `eb-blue`) dan sort key `timestamp` dalam format ISO 8601. Dua Global Secondary Index (GSI) ditambahkan: `log-level-index` untuk query log berdasarkan level (INFO, WARN, ERROR) dan `request-id-index` untuk distributed tracing yang memungkinkan penelusuran satu request melintas beberapa service. **Tabel `gb-app-prod-api-requests`** mencatat detail setiap request API yang masuk: endpoint yang dituju, HTTP method, status code response, durasi eksekusi dalam milidetik, IP address client, user agent, dan pesan error jika ada — data ini sangat berharga untuk analisis performa dan debugging. **Tabel `gb-app-prod-deployments`** mencatat riwayat setiap deployment yang dilakukan: siapa yang deploy, kapan, ke environment mana (blue/green), versi berapa, dan apakah berhasil. Ketiga tabel menggunakan billing mode `PAY_PER_REQUEST` (sebelumnya disebut On-Demand) yang lebih cocok untuk workload yang tidak konsisten, dan semuanya dikonfigurasi dengan **TTL (Time-to-Live)** pada atribut `expire_at` sehingga log lama dihapus otomatis oleh DynamoDB tanpa biaya tambahan.

### 10. Lambda Functions

Lambda adalah layanan komputasi serverless yang mengeksekusi kode hanya ketika ada request, tanpa server yang perlu dikelola. Dalam arsitektur ini, dua Lambda function dibuat untuk menangani request dari API Gateway. **Lambda POST Handler** (`gb-app-prod-api-post`) menangani semua operasi yang memodifikasi data: POST (membuat item baru), PUT (memperbarui item), dan DELETE (menghapus item). Setiap operasi diawali dengan logging request masuk ke DynamoDB, dilanjutkan dengan validasi input, eksekusi query ke RDS dalam transaction untuk menjaga konsistensi data, dan diakhiri dengan logging response keluar beserta durasi eksekusi. **Lambda GET Handler** (`gb-app-prod-api-get`) menangani operasi baca: GET list items dengan pagination, GET item by ID, dan GET health check. Health check endpoint secara aktif menguji koneksi ke database dan melaporkan status yang akurat. Kedua Lambda function berjalan di dalam VPC (Private Subnet) agar dapat mengakses RDS yang juga berada di Private Subnet. Keduanya menggunakan **Lambda Layer** yang berisi shared utilities: `dynamodb-logger.js` (utilitas logging ke DynamoDB), `db-connection.js` (koneksi pool ke MySQL dengan credential dari Secrets Manager), dan `response-helper.js` (standardisasi format response API). Konfigurasi runtime menggunakan Node.js 20.x dengan memory 128MB (optimal untuk free tier) dan timeout 30 detik. X-Ray tracing diaktifkan untuk distributed tracing.

### 11. API Gateway

API Gateway berfungsi sebagai pintu masuk (entry point) tunggal untuk semua request API dari client. Ia menerima request HTTP, memvalidasinya, lalu meneruskannya ke Lambda function yang sesuai menggunakan integrasi `AWS_PROXY`. REST API yang dibuat memiliki dua resource utama: `/items` (untuk operasi koleksi) dan `/items/{id}` (untuk operasi per item). Method yang tersedia mencakup `GET /items` untuk mengambil daftar item, `GET /items/{id}` untuk mengambil item tertentu, `POST /items` untuk membuat item baru, `PUT /items/{id}` untuk memperbarui item, dan `DELETE /items/{id}` untuk menghapus item. Semua method GET diarahkan ke Lambda GET Handler, sementara POST/PUT/DELETE diarahkan ke Lambda POST Handler. Sebuah endpoint `GET /health` yang mengembalikan response mock (tanpa memanggil Lambda) disediakan untuk health check ringan tanpa biaya komputasi. API Gateway dikonfigurasi sebagai **REGIONAL** endpoint (bukan EDGE) karena lebih murah dan sudah cukup untuk deployment di satu region. **Stage** bernama `prod` dibuat dengan access logging yang mencatat setiap request ke CloudWatch Logs — format log mencakup request ID, IP sumber, waktu, HTTP method, path, status code, dan durasi. **Usage Plan** dibuat dengan batas kuota 1 juta request per bulan dan throttling 1000 req/detik, selaras dengan batas Free Tier AWS.

### 12. AWS Amplify

Amplify adalah layanan hosting frontend yang terintegrasi penuh dengan GitHub, memungkinkan proses build dan deploy frontend berjalan otomatis tanpa konfigurasi server. Amplify terhubung ke GitHub repository melalui Personal Access Token, dan dikonfigurasi untuk memonitor branch `main`. Setiap kali ada push ke branch tersebut, Amplify secara otomatis menjalankan proses build sesuai spesifikasi di `build_spec`: menginstal dependency dengan `npm ci`, menjalankan `npm run build`, lalu mendeploy output dari direktori `dist/` ke CDN global AWS. Amplify dikonfigurasi dengan **SPA (Single Page Application) routing** — semua path yang tidak ditemukan akan diarahkan ke `index.html`, memungkinkan client-side routing bekerja dengan benar di framework seperti React, Vue, atau Angular. Environment variable seperti `VITE_API_URL` (URL API Gateway) dan `VITE_APP_NAME` diinject saat proses build agar frontend mengetahui ke mana harus mengirim request. Fitur **Pull Request Preview** diaktifkan sehingga setiap PR yang dibuka akan otomatis mendapat URL preview tersendiri untuk code review. Sebuah **Amplify Webhook** dibuat untuk memungkinkan GitHub Actions men-trigger rebuild Amplify secara manual jika diperlukan.

### 13. GitHub Actions (CI/CD Pipeline)

GitHub Actions adalah mesin otomasi yang mengeksekusi pipeline CI/CD setiap kali ada event di repository GitHub. Workflow yang dibuat (`blue-green-deploy.yml`) terdiri dari tiga job yang berjalan secara berurutan. **Job pertama (test-and-build)** bertugas memastikan kualitas kode: menginstal dependency, menjalankan unit test lengkap dengan laporan coverage, lalu mem-build aplikasi untuk production. Jika test gagal, seluruh pipeline berhenti dan deployment tidak terjadi. Setelah build berhasil, aplikasi di-package menjadi file ZIP dengan mengecualikan file-file yang tidak diperlukan (node_modules, test files, dll), dan ZIP tersebut diupload sebagai GitHub artifact. Nama versi dibuat dinamis berdasarkan warna environment, timestamp, dan 8 karakter pertama dari Git SHA untuk traceability. **Job kedua (deploy)** mengambil artifact ZIP, mengunggahnya ke S3 deployment bucket dengan metadata yang kaya (git SHA, warna deployment, nama deployer), membuat Elastic Beanstalk Application Version baru, lalu menerapkannya ke environment yang sesuai (GREEN untuk branch `main`, BLUE untuk branch `release/*`). Setelah deployment selesai, dilakukan health check dengan retry otomatis. Jika berhasil, record deployment dicatat ke tabel DynamoDB. **Job ketiga (switch-traffic)** hanya berjalan jika workflow dipicu secara manual dengan parameter `swap_traffic = true`. Job ini mengambil ARN Load Balancer dan kedua Target Group, lalu memodifikasi listener ALB untuk memindahkan 100% traffic ke environment yang baru di-deploy, menyelesaikan proses Blue/Green switch.

## Alur Kerja Blue/Green Deployment

Ketika seorang developer selesai mengembangkan fitur baru dan melakukan push ke branch `main`, seluruh mesin otomasi mulai bergerak. GitHub Actions terbangun, menjalankan tes, membangun aplikasi, dan menghasilkan paket deployment bernama misalnya `green-20260429154530-a1b2c3d4.zip`. Paket ini diunggah ke S3, versi baru dibuat di Elastic Beanstalk, dan deployment dijalankan ke environment GREEN yang saat ini dalam kondisi standby. Environment GREEN menerima versi baru, instance lama digantikan secara rolling, dan health check memverifikasi bahwa aplikasi baru berjalan dengan benar.

Pada titik ini, environment BLUE masih melayani 100% traffic produksi, sementara GREEN sudah menjalankan versi terbaru namun belum menerima traffic dari pengguna. Tim QA dapat mengakses URL environment GREEN secara langsung untuk melakukan pengujian akhir tanpa mempengaruhi pengguna produksi. Jika semua pengujian lulus, seorang deployment engineer menjalankan perintah `terraform apply -var="active_color=green"` atau men-trigger GitHub Actions dengan parameter `swap_traffic=true`. Dalam hitungan detik, ALB memindahkan seluruh traffic dari environment BLUE ke environment GREEN. Pengguna tidak merasakan gangguan apapun. Jika kemudian ditemukan masalah serius pada versi baru, rollback dilakukan dengan membalik perintah tersebut — traffic kembali ke BLUE dalam hitungan detik, tanpa perlu re-deploy ulang.

## Persyaratan Pengerjaan

Mahasiswa diminta untuk mengimplementasikan seluruh arsitektur yang telah dijelaskan di atas menggunakan Terraform sebagai Infrastructure as Code. Seluruh konfigurasi infrastruktur harus dinyatakan dalam file `.tf` yang terorganisir dengan baik berdasarkan fungsinya. Kode Lambda function harus ditulis dengan lengkap termasuk mekanisme logging ke DynamoDB di setiap titik eksekusi yang signifikan. Workflow GitHub Actions harus mencakup seluruh tiga tahap (test, deploy, switch-traffic) dengan pencatatan deployment ke DynamoDB. Seluruh implementasi harus memperhatikan batasan Free Tier AWS agar biaya operasional dapat diminimalkan. Dokumentasi berupa README.md yang komprehensif harus disertakan, mencakup penjelasan arsitektur, panduan langkah demi langkah untuk deployment, dan prosedur operasional Blue/Green switching serta rollback.
